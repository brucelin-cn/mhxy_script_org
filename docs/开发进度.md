### 当前结论

- 第一版应做成独立调度入口，而不是继续堆到 `mhxy_richang.py`
- 多窗口第一版采用串行轮询，不做真并发
- 首批优先接入 `baotu/mijing/dati/yabiao/shimeng`
- 配置建议使用 JSON
- 现阶段已具备“配置加载 -> 会话轮询 -> 任务适配 -> 基础恢复”的命令行闭环
- 旧自动化环境依赖仍然较重，但已不再阻塞调度器入口本身启动
- 调度器现在可以显式报告运行环境缺包，而不需要等到任务导入时才报错
- 首批任务适配现在已通过模块导入验证，当前只剩 `playsound` 作为可选能力缺失
- 项目现在已有可独立执行的游戏进程启动入口，不必依赖 UI 间接拉起
- 当前机器上的游戏安装路径配置已正确，真实启动阻塞点已收敛到权限要求
- 当前启动链路已细化为：
  - `launcher` 可启动并可检测到进程
  - `game exe` 可直接提权启动并可检测到进程
  - 真实图像识别仍受 OpenCV 缺失限制
- 启动策略需要以官方 launcher 为主，不能依赖多开 launcher 或直接常态启动 game exe
