# 调度器设计方案

## 1. 目标

本方案用于为当前仓库设计一个可落地的“多窗口任务调度器”，满足以下目标：

- 多窗口运行
- 每个窗口执行同一套任务序列
- 任务失败后不中断整体
- 大部分任务完成条件基于图片匹配
- 支持按轮次、按时间、按资源耗尽停止
- 接入现有 PyQt UI

第一版目标不是替换所有老脚本，而是在现有仓库之上增加一层稳定的调度能力。

## 2. 第一版范围

### 2.1 第一版接入任务

第一版优先接入日常类任务：

- `baotu`
- `mijing`
- `dati`
- `yabiao`
- `shimeng`

原因：

- 这些任务已有独立入口
- 与 `mhxy_richang.py` 聚合关系清晰
- 完成判断大多可通过图片匹配实现
- 比抓鬼、挖矿、购物类任务更适合先统一调度

### 2.2 第一版暂不覆盖

- `ghost`
- `fuben`
- `bangpai2`
- `mine`
- `shopping*`
- 远程控制链路
- 自动重登全闭环
- 真正并发多窗口执行

## 3. 总体架构

建议新增一个独立入口脚本：

- `mhxy_scheduler.py`

建议新增一个独立配置目录：

- `resources/scheduler/`

建议新增一组调度相关模块：

```text
scheduler/
├─ __init__.py
├─ config.py
├─ models.py
├─ task_specs.py
├─ adapters.py
├─ recovery.py
├─ runner.py
└─ reporter.py
```

说明：

- `mhxy_scheduler.py`
  - 命令行入口
- `scheduler/config.py`
  - 读取调度配置
- `scheduler/models.py`
  - 核心数据结构
- `scheduler/task_specs.py`
  - 任务定义与注册
- `scheduler/adapters.py`
  - 对接现有任务脚本
- `scheduler/recovery.py`
  - 恢复动作封装
- `scheduler/runner.py`
  - 调度主循环
- `scheduler/reporter.py`
  - 状态输出与 UI 可读信息

## 4. 设计原则

- 调度层不直接写死某个任务内部流程
- 任务层只负责“执行”和“完成判断”
- 恢复逻辑从任务逻辑中剥离
- 多窗口第一版采用串行轮询，不做真并发
- UI 只做入口与展示，不承载调度核心逻辑

## 5. 运行模型

### 5.1 核心模型

调度器由三层组成：

- 全局调度器
- 窗口会话
- 任务适配器

### 5.2 执行方式

第一版使用：

- 单进程
- 多窗口串行轮询
- 每次只激活一个窗口

原因：

- 当前底层依赖全局 `frame/frameSize/frameSizeCm`
- 并行执行会非常容易互相污染
- 串行轮询更适合当前代码基线

### 5.3 调度主流程

```text
启动调度器
-> 读取配置
-> 构建窗口会话
-> 初始化任务定义
-> 进入主循环
   -> 检查全局停止条件
   -> 轮询窗口
      -> 跳过已暂停/已完成窗口
      -> 选择当前任务
      -> 执行任务
      -> 判断完成/失败/超时
      -> 更新窗口状态
      -> 必要时恢复或切换任务
-> 所有窗口结束后退出
```

## 6. 核心数据结构

### 6.1 SchedulerConfig

用于表达调度器配置。

建议字段：

```python
class SchedulerConfig:
    window_indices: list[int]
    mission_order: list[str]
    stop_mode: str
    round_limit: int | None
    time_limit_minutes: int | None
    resource_limit_enabled: bool
    poll_interval_seconds: float
    max_task_failures: int
    max_window_failures: int
    enable_recovery_level_1: bool
    enable_recovery_level_2: bool
    enable_recovery_level_3: bool
    enable_screenshots: bool
    enable_ui_reporting: bool
```

### 6.2 WindowSession

用于表达单窗口运行状态。

建议字段：

```python
class WindowSession:
    window_index: int
    current_task: str | None
    current_task_index: int
    rounds_completed: int
    failures_in_row: int
    task_failures: dict[str, int]
    paused: bool
    completed: bool
    last_success_at: datetime | None
    last_error: str | None
    last_state: str | None
```

### 6.3 TaskSpec

用于表达一个任务的定义。

建议字段：

```python
class TaskSpec:
    name: str
    adapter_name: str
    done_images: list[str]
    timeout_seconds: int
    recovery_levels: list[str]
    enabled: bool = True
```

### 6.4 TaskResult

用于表达单次任务执行结果。

建议字段：

```python
class TaskResult:
    status: str
    state_name: str | None
    message: str | None
    done: bool
    should_retry: bool
    should_skip: bool
    should_pause_window: bool
```

状态建议值：

- `success`
- `soft_fail`
- `recoverable_fail`
- `hard_fail`
- `timeout`

## 7. 任务定义

### 7.1 第一版统一任务序列

所有窗口使用同一套任务序列：

```text
baotu -> mijing -> dati -> yabiao -> shimeng
```

### 7.2 任务适配方式

第一版建议不要直接重写这些任务的内部实现，而是先做适配器层。

每个任务适配器统一暴露：

- `prepare(window_session)`
- `run(window_session)`
- `check_done(window_session)`
- `recover(window_session, level)`

### 7.3 第一版适配策略

建议优先使用“类调用适配”，而不是“子进程脚本调用适配”。

优先原因：

- 更容易拿到执行结果
- 更容易做窗口级状态更新
- 更容易做恢复

但由于现有脚本结构不完全统一，第一版允许混合策略：

- 能直接类调用的，优先类调用
- 暂时难以类调用的，保留脚本入口作为过渡

## 8. 完成条件设计

### 8.1 完成条件类型

统一支持三类完成条件：

- `round_limit`
- `time_limit`
- `resource_limit`

### 8.2 任务级完成判断

第一版每个任务至少配置：

- 主完成图片
- 任务超时

完成判断优先级建议：

1. 先检查完成图片
2. 再检查任务是否超时
3. 再检查外部停止条件

### 8.3 调度级完成判断

全局停止判断支持：

- 达到总轮次
- 达到总运行时长
- 资源耗尽
- 所有窗口已完成
- 所有窗口已暂停

## 9. 恢复策略

### 9.1 恢复分级

#### Level 1: 通用轻恢复

- 关闭通用弹窗
- 重点任务追踪
- 再点确认/继续/领取
- 重进活动入口

#### Level 2: 窗口级恢复

- 重新激活窗口
- 重新初始化窗口上下文
- 重新切换窗口大小
- 返回主界面后重进任务

#### Level 3: 会话级恢复

- 断线重登
- 跳过当前任务
- 暂停当前窗口

### 9.2 第一版推荐策略

- 单次失败：重试
- 连续失败达到阈值：执行 `Level 1`
- `Level 1` 无效：执行 `Level 2`
- `Level 2` 无效：跳过当前任务
- 当前窗口连续多个任务失败：暂停该窗口

### 9.3 恢复矩阵

#### `baotu`

- 完成判断
  - 完成图 + 任务超时
- 允许恢复
  - Level 1
  - Level 2

#### `mijing`

- 完成判断
  - 完成图 + 任务超时
- 允许恢复
  - Level 1
  - Level 2

#### `dati`

- 完成判断
  - 完成图 + 任务超时
- 允许恢复
  - Level 1

#### `yabiao`

- 完成判断
  - 完成图 + 任务超时
- 允许恢复
  - Level 1
  - Level 2

#### `shimeng`

- 完成判断
  - 完成图 + 任务超时
- 允许恢复
  - Level 1
  - Level 2

## 11. 与现有代码的接点

### 11.1 现有调度入口

当前最接近调度器的入口是：

- `mhxy_richang.py`

但它的问题是：

- 是脚本式串行
- 缺少状态对象
- 缺少失败分级
- 缺少统一恢复策略

所以第一版不建议直接在这个文件里继续叠复杂逻辑。

### 11.2 现有 UI 入口

当前入口是：

- `ui/mhxy_pyqt.py`

它的问题是：

- 通过命令字符串启动脚本
- 多窗口选择逻辑不完整
- 无法展示真实运行状态

所以接入调度器后，建议新增“调度器启动入口”，而不是继续复用现有 `richangTask()` 的语义。

### 11.3 现有底层能力

可优先复用：

- `mhxy/__init__.py`
- `gotoActivity()`
- `Util.locateCenterOnScreen()`
- `Util.locateOnScreen()`
- `allEscapeTeam()`

需要注意：

- `init()` 依赖全局状态
- 多窗口切换时必须重新初始化上下文

## 12. UI 接入方案

### 12.1 第一版 UI 目标

第一版 UI 至少提供：

- 选择是否启用调度器
- 选择窗口范围
- 选择任务序列
- 启动调度
- 停止调度
- 查看每个窗口当前任务状态

### 12.2 建议新增 UI 区域

建议在现有 PyQt 页面新增“调度器”区域，包含：

- 任务复选框
- 窗口选择
- 总运行时长输入
- 总轮次输入
- 启动按钮
- 停止按钮
- 状态面板

### 12.3 第一版不要做的 UI

- 不做复杂实时图表
- 不做远程控制面板
- 不做历史运行报表

## 13. 日志与监控

第一版建议额外输出：

- 当前窗口
- 当前任务
- 当前状态名
- 当前轮次
- 最近失败原因
- 连续失败次数

建议保留未来可扩展能力：

- 失败截图
- UI 状态刷新
- JSON 状态导出

## 14. 验证方案

### 14.1 文档级验证

- 检查设计文档和现有项目结构是否一致
- 检查第一版任务是否都已有现成入口
- 检查配置草案是否能覆盖当前规划要求

### 14.2 开发前验证

真正开始编码前建议先做：

- 确认 `baotu/mijing/dati/yabiao/shimeng` 的适配方式
- 确认哪些任务能直接类调用
- 确认 UI 当前是否已有可复用控件区域

### 14.3 编码后验证

第一版实现后建议按这个顺序验证：

1. 单窗口 + 单任务
2. 单窗口 + 多任务
3. 多窗口 + 同构任务序列
4. 单任务失败后跳过
5. 单窗口连续失败后暂停
6. UI 启动与停止

## 15. 第一版开发顺序

1. 新建调度器设计相关文档和进度记录
2. 定义配置结构和核心模型
3. 落命令行版调度器骨架
4. 先适配 `baotu` 和 `mijing`
5. 补状态输出和失败分级
6. 扩展到 `dati/yabiao/shimeng`
7. 接入 PyQt UI
8. 再补恢复矩阵和验证

## 16. 风险

- 当前底层全局状态会限制多窗口设计
- 现有部分脚本类接口不统一
- 部分完成图片需要二次确认
- UI 现状不适合直接承接复杂状态展示

## 17. 结论

第一版调度器应以“多窗口串行轮询 + 同构任务序列 + 图片驱动完成判断 + 失败跳过”为核心策略。

先把调度层抽出来，比继续在单脚本上叠循环更稳，也更符合后续扩展目标。
