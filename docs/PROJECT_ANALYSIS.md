# mhxy_script_org 项目分析

## 1. 项目定位

这是一个面向 Windows 环境的 Python 自动化脚本项目，目标是为《梦幻西游：时空》相关 PC/桌面端窗口提供半自动或全自动操作能力。

项目的核心思路不是调用游戏 API，而是基于以下手段做界面级自动化：

- 通过窗口标题获取目标游戏窗口
- 基于截图模板做界面元素识别
- 将固定坐标和相对坐标映射到窗口内
- 通过鼠标点击、拖拽、键盘输入完成任务流程

从代码组织看，这个仓库已经不是单脚本集合，而是一个“自动化内核 + 多个任务脚本 + PyQt 控制台 + 图片资源库”的完整工具仓库。

## 2. 技术栈

后端/脚本层：

- Python 3.8+
- `PyAutoGUI`：截图、窗口、鼠标键盘控制
- `pydirectinput`：键盘输入补充
- `opencv-contrib-python`：模板匹配置信度支持
- `Pillow` / `numpy`：图像相关依赖
- `playsound`：异常或超时提醒
- `psutil`：进程探测、启动/停止游戏
- `requests`：部分远程控制场景依赖

桌面 UI：

- `PyQt6`
- `qtpy`

运行平台特征：

- 明显偏向 Windows
- 依赖桌面窗口、屏幕分辨率、焦点切换
- 对 Mac/Linux 基本没有兼容设计

## 3. 目录结构

```text
.
├─ mhxy/                    基础运行时与通用工具
├─ resources/               各任务用到的截图模板、配置、说明文件
├─ ui/                      PyQt 图形界面
├─ game_process.py          窗口缩放/排布工具
├─ game_conf.py             游戏程序路径配置读取
├─ game_exehelp.py          游戏进程启动/停止辅助
├─ game_watcher.py          断线/重登监控雏形
├─ pic_target.py            图片目标与点击策略抽象
├─ mhxy_*.py                各类任务脚本
└─ README.md                现有说明，当前存在编码可读性问题
```

## 4. 核心架构

### 4.1 运行内核：`mhxy/__init__.py`

这是全项目最核心的文件，承担了大部分基础设施职责：

- 日志初始化，输出到 `mhxy_script.log`
- 全局窗口框架定义：`frame`、`frameSize`、`originSize`、`smallSize`
- 窗口内相对坐标换算
- 基于图片的查找与点击封装
- 战斗中/战斗后流程辅助
- 任务跳转、活动面板打开、背包使用等公共动作
- `MhxyScript` 基类，用于统一窗口初始化和停止控制

这个文件本质上相当于项目的“自动化 SDK”。

### 4.2 目标模型：`pic_target.py`

这个模块把“识别图片并点击”的行为抽象成结构化对象：

- `PicTarget`：描述一张图、识别方式、点击方式
- `PicLocateType`：整块定位或中心点定位
- `PicClickType`：按右下角、按中心点或自定义偏移点击
- `tryClickPic()`：识别成功后执行点击

这让 `mhxy_common.py` 和部分任务脚本可以用声明式方式配置流程。

### 4.3 窗口管理：`game_process.py`

这个脚本用于统一游戏窗口布局，方便多开和脚本坐标复用：

- 将窗口缩放到 `smallSize=(907, 707)` 或还原到 `originSize=[1040, 807]`
- 横向/纵向排列多个窗口
- 通过索引只调整某个目标窗口

它是很多任务脚本能稳定运行的前置条件。

### 4.4 配置与进程管理

相关文件：

- `game_conf.py`
- `game_exehelp.py`
- `game_watcher.py`

职责分别是：

- 从 `resources/common.ini` 读取游戏 EXE 路径、启动器路径、玩家配置
- 启动/停止游戏进程
- 监测断线并尝试重登

其中 `game_watcher.py` 更像“监控雏形”，有一定设计意图，但实现还不够完整。

### 4.5 图形界面：`ui/`

`ui/mhxy_pyqt.py` 是主入口，功能包括：

- 提供按钮式启动各类脚本
- 配置不同任务的参数弹窗
- 选择目标窗口
- 调整游戏窗口大小
- 打开日志
- 执行自定义脚本

UI 本质上是脚本启动器，而不是业务逻辑核心。

## 5. 业务脚本分层

## 6. 执行流程模型

从代码抽象上看，项目大多数脚本都遵循同一套路：

1. 通过 `init(idx=...)` 绑定某个游戏窗口
2. 读取窗口尺寸，建立窗口内坐标系
3. 根据任务需要选择是否缩放到小窗口
4. 通过 `gotoActivity()` 或固定入口进入任务
5. 循环查找图片资源
6. 按图片类型和点击策略执行动作
7. 用战斗状态、完成标识、超时机制决定是否继续

这套模型的优点是复用度高，缺点是对界面变化非常脆弱。

## 8. 资源组织方式

`resources/` 是整个仓库的核心资产之一。

目录按任务拆分，例如：

- `resources/common/`：通用按钮、登录、关闭、确认等
- `resources/richang/`：活动入口相关
- `resources/ghost/`：抓鬼任务

这些图片模板直接决定脚本稳定性。仓库中的代码和资源是强耦合关系，不能只迁移代码不迁移资源。

## 10. 当前项目的优点

- 自动化基础能力比较集中，复用性不错
- 资源目录划分清楚，便于按任务维护模板
- 基本具备多开窗口支持
- 有图形界面，降低了非代码用户使用门槛
- 配置文件外置，部分参数无需改代码
- 已经形成“通用能力 + 任务脚本”的雏形

## 11. 当前项目的主要问题

### 11.1 文档与编码问题

- 仓库现有 `README.md` 存在明显编码可读性问题
- 部分中文字符串在代码输出中也有乱码痕迹
- 新接手者很难仅靠现有说明完成环境搭建

### 11.2 全局状态过多

`mhxy/__init__.py` 使用了大量全局变量：

- `frame`
- `frameSize`
- `frameSizeCm`
- 日志对象

这使得：

- 多窗口并行时容易相互影响
- 模块测试困难
- 调用顺序必须非常谨慎

### 11.3 代码耦合较高

主要表现为：

- 业务脚本直接依赖图片路径与硬编码坐标
- 任务逻辑、界面识别、等待策略写在同一层
- 很少有可复用的状态机抽象

一旦游戏 UI 改版，维护成本会比较高。

### 11.4 错误处理不统一

虽然项目有日志和声音提醒，但整体仍偏“脚本式容错”：

- 很多地方只循环等待，没有明确失败分支
- 异常恢复逻辑零散
- 断线重连能力有雏形，但没有形成统一机制

### 11.5 命名与一致性问题

例子：

- `mhxy_shopping.py` 存在，但 UI 中调用的是 `mhxy_shopping1`
- `getTarget()` 中窗口索引映射存在跳号现象
- `game_watcher.py`、`game_exehelp.py` 中有部分返回值/流程不完整
- 同类任务脚本参数风格不完全统一

这些问题会增加后续维护成本。

### 11.6 大日志文件未忽略

仓库根目录存在 `mhxy_script.log`，且体积很大。

这说明：

- 运行日志可能被直接纳入工作区
- `.gitignore` 目前没有覆盖日志文件
- 长期会影响仓库体积与协作体验

## 12. 维护建议

按投入产出比排序，建议优先做下面几件事。

### 12.1 先修文档和仓库卫生

- 修复 `README.md` 编码
- 补一份清晰的使用说明
- 把 `mhxy_script.log`、临时截图、打包产物加入 `.gitignore`

### 12.2 把“窗口上下文”对象化

建议把下面这类全局状态收敛到单个上下文对象中：

- 窗口句柄
- frame 坐标
- frameSize
- 配置

这样可以明显降低多窗口互相污染的风险。

### 12.3 把识别流程抽成状态机

当前很多任务是“无限循环 + if/else 查图点击”。

后续可以拆成：

- 进入任务
- 对话处理
- 战斗等待
- 领取奖励
- 结束退出

这样脚本会更稳定，也更容易诊断失败在哪一步。

### 12.4 统一脚本入口协议

建议所有脚本统一参数风格，例如：

- `-i / --idx`
- `-ir / --idx-array`
- `-m / --mission`
- `--resize`
- `--shutdown`

统一后，UI 和 CLI 都更容易维护。

### 12.5 对资源做版本化管理

建议为关键模板建立最基本的说明：

- 这个图对应什么界面
- 建图时窗口尺寸是多少
- 推荐识别阈值是多少
- 如果失效应该在哪个流程重新截图

这会显著降低“资源失效但没人知道该怎么修”的问题。

## 13. 适合接手者的阅读顺序

如果后续要继续维护，建议按下面顺序读代码：

1. `mhxy/__init__.py`
2. `pic_target.py`
3. `game_process.py`
4. `mhxy_richang.py`
5. `mhxy_baotu.py`、`mhxy_ghost.py`、`mhxy_fuben.py`
6. `ui/mhxy_pyqt.py`

这个顺序能先理解“底层能力”，再看“具体任务”，最后看“UI 如何调度脚本”。

## 14. 结论

这是一个已经沉淀出实际可用能力的游戏界面自动化项目，不是简单 demo。

它的强项在于：

- 对具体任务场景积累了较多图片资源和脚本经验
- 具备多脚本、多任务、多窗口协同基础
- 有 UI 包装，实际可操作性强

它的短板也很明确：

- 代码偏脚本化，工程化不足
- 全局状态重
- 对分辨率、窗口排布、图片模板高度敏感
- 文档和一致性还有较大提升空间

如果后续目标是“继续个人使用”，当前结构还能支撑一段时间。
如果后续目标是“长期维护或交接”，优先级最高的工作是补文档、收敛全局状态、统一脚本协议。

## 15. 代码分层的真实情况

虽然仓库表面上分成了 `mhxy/`、`ui/`、`resources/` 和若干 `mhxy_*.py` 脚本，但从调用关系看，当前项目更接近下面这 5 层。

### 15.1 第 0 层：运行环境假设

这层没有显式代码文件，但几乎所有脚本都依赖它：

- 必须有可见的 Windows 桌面窗口
- 游戏窗口标题要能被 `getWindowsWithTitle()` 找到
- 窗口尺寸需要接近 `originSize` 或 `smallSize`
- 屏幕缩放、字体缩放、UI 皮肤变化不能太大
- 鼠标和键盘输入不能被其他程序抢占

这意味着很多 bug 并不是“代码逻辑错误”，而是“运行前提不成立”。

### 15.2 第 1 层：底层运行时

文件：

- `mhxy/__init__.py`
- `pic_target.py`

职责：

- 窗口发现
- 坐标换算
- 图片识别
- 点击/输入
- 日志
- 战斗前后辅助逻辑

这一层的特点是“全局状态很重”。上层任何一个脚本出问题，最终大概率都要回到这里看。

### 15.3 第 2 层：通用任务引擎

文件：

- `mhxy_common.py`

它试图把“任务执行”抽象成一套通用流程：

- 任务栏识别
- 道具使用
- NPC 交互
- 过场处理
- 战斗处理
- 特殊逻辑
- 完成标记

`mhxy_shimeng.py` 已经开始复用这套能力，但仓库里大多数老脚本仍然是各自写死流程，没有统一接到这层。

### 15.4 第 3 层：业务脚本

文件：

- `mhxy_ghost.py`
- `mhxy_fuben.py`
- `mhxy_baotu.py`
- `mhxy_mijing.py`
- `mhxy_dati.py`
- `mhxy_yabiao.py`
- `mhxy_bangpai2.py`
- `mhxy_mine.py`
- 其他 `mhxy_*.py`

这层是真正实现任务行为的地方。大多数 bug 也集中在这里：

- 找图条件不对
- 点击顺序不对
- 未处理某个弹窗
- 超时逻辑缺失
- 战斗/非战斗状态切换判断不稳

### 15.5 第 4 层：调度与入口层

文件：

- `ui/mhxy_pyqt.py`
- `mhxy_richang.py`
- `mhxy_520.py`
- `game_process.py`
- `game_exehelp.py`
- `game_watcher.py`

这层负责：

- 启动脚本
- 串联多个任务
- 调整窗口
- 进程管理
- 登录恢复

它本来应该是“编排层”，但目前也混有不少具体业务判断。

## 16. 关键调用链

后续修 bug 时，最重要的是先判断 bug 落在哪条链路上。

### 16.1 UI 触发链

典型路径：

1. `ui/mhxy_pyqt.py` 按钮点击
2. `exec_script()` 组装命令
3. `os.system("start python[w] ...")`
4. 对应 `mhxy_*.py` 启动
5. 脚本内部调用 `init()`
6. 再进入具体任务循环

这条链的问题是：

- UI 和脚本靠命令字符串拼接耦合
- UI 不知道脚本执行成功还是失败
- 无法拿到脚本实时状态

### 16.2 单任务执行链

典型路径：

1. 入口脚本解析参数
2. `MhxyScript.__init__()` 调用 `init()`
3. `init()` 设置全局 `frame/frameSize/frameSizeCm`
4. 任务脚本调用 `Util.locateCenterOnScreen()` / `Util.locateOnScreen()`
5. 识别成功后执行 `pyautogui` 点击
6. 根据下一张图或下一种状态继续

这条链是仓库最核心的主路径。

### 16.3 聚合任务链

典型路径：

1. `mhxy_richang.py` 或 `mhxy_520.py`
2. 通过 `gotoActivity()` 进入活动
3. 构造子任务对象
4. 复用上一阶段的 `config` 或重新 `init()`
5. 串行跑多个脚本

这条链的风险主要在“上下文串扰”，尤其是多任务连跑时：

- 前一个任务可能留下弹窗
- 窗口状态被改变
- `config` 里的 frame 信息被复用

## 17. 高风险 bug 热点

下面这些不是抽象风险，而是从代码上看就比较像真实缺陷或高概率缺陷点。

### 17.1 UI 目标窗口索引映射不一致

文件：

- `ui/mhxy_pyqt.py`

问题：

- `getTargetArr()` 返回的是 `0,1,2,3,4`
- `getTarget()` 返回的是 `0,1,2,4,5`

影响：

- 单窗口任务和多窗口任务可能打到不同窗口
- 第 4、第 5 个窗口的行为会和预期不一致

这类问题属于“用户点的是 A，脚本打的是 B”，优先级很高。

### 17.2 多选逻辑名义支持，实际不支持

文件：

- `ui/mhxy_pyqt.py`

问题：

- `allChk()` 会把多个目标勾上
- 但 `getTargetArr()` 用的是 `if / elif`
- 最终只会返回第一个被勾中的窗口

影响：

- UI 让用户误以为支持多选
- 实际脚本只执行一个目标

### 17.3 `jingyanlianTask()` 调错脚本

文件：

- `ui/mhxy_pyqt.py`

问题：

- “经验链”按钮实际执行的是 `mhxy_yabiao`

影响：

- 这是直接的功能错误
- 用户从 UI 上完全看不出来

### 17.4 `shopping1Task()` 指向不存在或不一致脚本名

文件：

- `ui/mhxy_pyqt.py`

问题：

- UI 调的是 `mhxy_shopping1`
- 仓库里看到的是 `mhxy_shopping.py`

影响：

- 按钮可能根本无法启动
- 或依赖外部未纳入仓库的脚本

### 17.5 `GameExeHelp.hasExe()` 没有返回值

文件：

- `game_exehelp.py`

问题：

- `hasExe()` 调用了 `hasExe_()` 但没有 `return`

影响：

- `game_watcher.py` 中 `if help.hasExe():` 实际永远拿到 `None`
- 断线监控逻辑分支几乎不可能按预期工作

### 17.6 `GameWatcher` 自身也有多处逻辑未完成

文件：

- `game_watcher.py`

问题：

- `getNextPlayer()` / `getCurPlayer()` 没有返回值
- 登录按钮资源名出现 `jinru_youxi.png.png`
- `while findPicNode is not None:` 里每次又把 `findPicNode = top`
- 整个流程存在明显未完成痕迹

影响：

- 这个模块目前更像草稿，不适合作为稳定能力依赖

### 17.7 `Common` 的旧接口与新引擎已经部分脱节

文件：

- `mhxy_common.py`

问题：

- `addTrigger()` 会往 `self.targets` 里加内容
- 但 `mission()` 现在只遍历 `opTypeTargetListMap`
- `self.targets` / `self.done_pics` 基本已经不生效

影响：

- 调用方以为加了 trigger，实际执行不到
- 很容易出现“代码写了但功能没接上”的隐蔽问题

### 17.8 `Common.Done` 阶段没有显式结束语义

文件：

- `mhxy_common.py`

问题：

- `Done` 只是普通的图片点击
- 找到“完成”图片后不会 `return` 或 `break`

影响：

- 某些任务即使结束了，也可能继续循环、继续找图、继续自动关闭弹窗

### 17.9 `mhxy_shimeng.py` 的 CLI 多窗口参数没有真正生效

文件：

- `mhxy_shimeng.py`

问题：

- 参数是 `idxArray`
- 最终只执行了 `func(int(indexArr[0]))`

影响：

- 看起来支持批量，实际只跑第一个窗口

### 17.10 全局窗口上下文会导致多任务串扰

文件：

- `mhxy/__init__.py`

问题：

- `frame`
- `frameSize`
- `frameSizeCm`

这些都是模块级全局变量。

影响：

- 多窗口串行脚本靠“最后一次 init 的结果”工作
- 任意一个任务重新 `init()`，都可能影响后续任务
- 想做并行或线程化时几乎必然出问题

### 17.11 `gotoActivity()` 和若干等待函数缺少统一超时退出

文件：

- `mhxy/__init__.py`

问题：

- `gotoActivity()` 会长期等待活动入口
- `waitUtilFindPic()` 也默认无限等
- 大量任务主循环是 `while True`

影响：

- 一旦图片失效、界面卡住、弹窗遮挡，脚本只会一直挂住
- 不利于定位是“识别失败”还是“逻辑失败”

### 17.12 日志量过大，且每次找图都打印

文件：

- `mhxy/__init__.py`

问题：

- `Util.locateCenterOnScreen()` 每次调用都 `log()`
- 仓库已有超大 `mhxy_script.log`

影响：

- 实际排查时高价值日志会被噪声淹没
- I/O 会拖慢运行
- 仓库会被日志污染

## 18. 架构性风险

这些问题短期不一定炸，但会直接影响后续新功能开发速度。

### 18.1 业务逻辑和识别策略混写

目前很多脚本把这些内容写在一起：

- 找哪张图
- 图出现说明什么状态
- 该点哪里
- 点完该等多久

结果是后续改一个任务时，往往要同时理解“识别”和“流程”两套逻辑。

### 18.2 坐标系是“隐式上下文”

调用 `Util.leftClick(7, 15)` 时，调用者默认依赖：

- 当前窗口已正确初始化
- 当前窗口尺寸匹配
- 当前 UI 布局和录制时一致

也就是说，代码本身看不到它依赖什么上下文，这会让排查 bug 很慢。

### 18.3 缺少统一任务生命周期

目前不同脚本停止的方式很多：

- `self._flag = False`
- `sys.exit(0)`
- `break`
- `return`
- 抛异常

结果是：

- 很难统一“停止任务”“清理状态”“退出前关闭弹窗”
- UI 也无法知道任务是“完成”“失败”还是“中断”

### 18.4 配置文件读写分散

现在每个任务脚本和每个配置弹窗都自己读取 ini：

- 键名没有统一校验
- 默认值没有集中定义
- 改名或迁移字段时容易漏

这会让配置越来越难维护。

### 18.5 网络控制能力没有边界

文件：

- `mhxy_remote_control.py`
- `mhxy_shopping2.py`
- `mhxy_remote_client.py`

现状：

- 端口写死
- 无鉴权
- 协议简单
- 出错恢复弱

如果未来要继续走远程调度路线，这块需要重新设计，不建议在现状上直接叠功能。

## 19. 新功能应该从哪里接入

后续开发新功能时，不建议“一上来就照着老脚本复制一份”。更稳的方式是先判断任务类型。

### 19.1 适合接到 `Common` 引擎的任务

条件：

- 任务主要由若干固定 UI 状态组成
- 每个状态可以靠图标判断
- 操作基本是点按钮、点任务栏、点确认

这类任务建议：

1. 新建 `resources/<task>/`
2. 新建 `mhxy_<task>.py`
3. 继承 `Common`
4. 通过 `addOpTypeTrigger()` 描述各类状态
5. 在 `mhxy_richang.py` 或 UI 中挂入口

典型例子：

- 师门
- 主线
- 日常里偏线性的交互任务

### 19.2 适合独立脚本实现的任务

条件：

- 任务包含复杂地图移动
- 需要特殊节奏控制
- 需要大量循环、战斗、回合判断

这类任务更适合保留独立逻辑，但建议复用底层公共函数。

典型例子：

- 抓鬼
- 挖矿
- 副本
- 抢购/蹲商品

### 19.3 适合先做“中间层抽象”再接功能的任务

如果你后面准备做以下功能，建议先重构一层再做：

- 任务队列
- 多窗口并行
- 失败自动恢复
- 任务调度面板
- 远程控制中心

因为这些能力会跨多个脚本，如果直接在现有脚本上打补丁，后面会越来越难收。

## 20. 建议的重构优先级

如果目标是“后续能稳定修 bug 和持续加功能”，我建议按下面顺序推进。

### 第一阶段：低风险高收益

- 修 UI 入口错误
- 统一 `.gitignore`
- 控制日志量
- 补 `README`
- 修 `GameExeHelp.hasExe()` 这类直接 bug

收益：

- 立刻降低误操作和调试成本

### 第二阶段：建立可维护边界

- 把窗口上下文封装成 `WindowContext`
- 把配置读写集中到统一模块
- 把任务结果统一为 `success / failed / interrupted / timeout`

收益：

- 后面再加任务时，不用每次都重新发明轮子

### 第三阶段：把业务逻辑做成可组合模块

建议抽出几类对象：

- `Detector`：负责识别图像
- `Action`：负责点击、输入、拖拽
- `TaskStep`：代表一个状态步骤
- `TaskRunner`：负责循环、超时、恢复

收益：

- 可以把“找图”和“业务流程”拆开
- 后续更容易做调试和回放

### 第四阶段：再考虑高级功能

- 多窗口并发
- 任务编排
- 远程队列
- Web/桌面监控面板

## 21. 调试和定位 bug 的建议方法

后面真要修 bug，建议不要一上来就改代码，可以按下面顺序判断。

### 21.1 先判断是哪一类问题

分类方法：

- 资源失效：图找不到
- 坐标失效：图找到了但点错
- 状态机失效：点对了但流程没往下走
- 入口层失效：UI 调错脚本、参数错、窗口错
- 环境失效：窗口尺寸、分辨率、焦点不对

### 21.2 复现时建议记录 4 个信息

- 当前运行的是哪个脚本
- 目标窗口索引是多少
- 当前窗口尺寸是多少
- 卡住时屏幕上实际显示了什么

如果只看日志，不看现场 UI，很多问题很难判断。

### 21.3 最值得先加的调试能力

如果后面要做稳定维护，我最建议先补这两个能力：

- 识别失败时可选保存截图
- 每一步输出“当前状态名”而不是只输出图片路径

这两个能力对定位 bug 的帮助会非常大。

## 22. 面向未来功能开发的建议结构

如果未来打算持续迭代，我建议把新代码逐步往下面这个结构靠拢：

```text
automation/
├─ runtime/         窗口、输入、截图、日志
├─ detectors/       图片模板与识别策略
├─ tasks/           各任务状态机
├─ configs/         配置加载与校验
├─ orchestrators/   日常串联、任务队列、远程调度
└─ ui/              图形界面
```

不需要一次重写，但新功能尽量按这个思路落，会比继续堆在根目录更稳。

## 23. 结论补充

如果只从“能不能跑”看，这个仓库已经有相当多沉淀。
但如果从“后续要长期修 bug 和开发新功能”看，真正的挑战已经不在单个任务脚本，而在下面三件事：

- 入口层是否可靠
- 全局上下文是否可控
- 任务流程是否能被清晰表达和调试

也就是说，后续最有价值的工作，不只是继续加图和加坐标，而是先让项目进入“可诊断、可演进”的状态。

## 24. 长循环任务规划清单

这一节不是具体实现方案，而是后续做“长时间循环执行任务”前，建议先明确的设计清单。

### 24.1 先定义任务边界

在开始做之前，建议先写清楚这 6 个问题：

- 任务目标是 修复代码的bug，优化代码，为后面新功能打基础
- 任务触发入口是什么
- 单次循环的开始条件是什么
- 单次循环的结束条件是什么
- 整个任务何时停止
- 任务失败后是重试、跳过还是终止

如果这 6 个问题没有先定下来，后面代码很容易变成“无限循环 + 碰到什么点什么”。

### 24.2 任务类型要先归类

在本项目里，长循环任务大致会落在 3 类之一：

- 纯状态轮询型
  - 例子：不断查某张图，出现就点击
- 流程推进型
  - 例子：进入任务、处理对话、战斗、领奖、回到入口，再开始下一轮
- 调度编排型
  - 例子：多个子任务按时间、优先级或资源条件循环执行

不同类型决定后面代码应该接在哪一层：

- 纯状态轮询型：适合独立脚本或工具脚本
- 流程推进型：优先考虑接入 `Common` 或做状态机脚本
- 调度编排型：更适合在 `mhxy_richang.py` 这一类调度入口之上实现

### 24.3 必须提前定义的输入

建议先明确这些输入项：

- 目标窗口范围
  - 单窗口还是多窗口
- 是否需要小窗模式
- 是否需要先进入某个活动/地图/NPC 对话
- 循环次数
  - 固定次数、无限次、按时间截止
- 轮询间隔
- 超时时间
- 失败重试次数
- 是否允许自动重登
- 是否允许自动关机

这些输入最好最终落到配置文件，而不是写死在脚本里。

### 24.4 必须提前定义的输出

长循环任务至少要有下面这些输出：

- 当前任务名
- 当前轮次
- 当前步骤/状态名
- 最近一次成功动作
- 最近一次失败原因
- 是否进入超时/重试/退出

如果后面需要排查问题，光知道“脚本还在跑”远远不够。

### 24.5 状态机建议先画出来

长循环任务在这个项目里，强烈建议先写状态图，再写代码。

最少建议先列出这些状态：

- `INIT`
- `PREPARE`
- `ENTER_TASK`
- `RUNNING`
- `BATTLE`
- `REWARD`
- `RECOVER`
- `RETRY`
- `EXIT`

然后为每个状态明确：

- 进入条件
- 退出条件
- 超时条件
- 失败转移去哪里

如果不先画状态，后面很容易在 `while True` 里面不断堆 `if/else`。

### 24.6 本项目里长循环最容易出问题的点

结合现有代码，长循环任务最容易踩的点主要有这些：

- 全局窗口上下文会被后续 `init()` 覆盖
- 某一步弹窗没处理，导致整条循环卡住
- 图片识别失败后没有超时出口
- 战斗状态和非战斗状态切换不稳定
- 多窗口时窗口索引和 UI 选择不一致
- 日志过大但缺少真正有用的状态信息
- 某次失败后没有恢复到可继续下一轮的初始状态

所以长循环任务不是“让它一直跑”这么简单，关键是“失败后能不能继续跑”。

### 24.7 恢复机制必须单独设计

长循环任务如果没有恢复机制，稳定性通常会很差。

建议至少提前定义这几类恢复动作：

- 关闭通用弹窗
- 重新点击任务追踪
- 重新打开活动面板
- 重新回到主界面/地图
- 检测断线并重登
- 触发人工告警并暂停

并且要定义每种恢复动作的触发条件，而不是只在出问题时临时补。

### 24.8 停止机制也要提前定义

建议先明确支持哪几种停止方式：

- 正常完成后停止
- 达到循环次数停止
- 达到运行时长停止
- 连续失败 N 次停止
- 检测到验证码/风控后停止
- 人工手动停止

对于长循环任务，停止不是附属功能，而是核心能力。

### 24.9 监控信息建议提前规划

如果任务要长时间运行，建议至少记录这些监控信息：

- 启动时间
- 已运行时长
- 已完成轮数
- 平均每轮耗时
- 最近一次成功时间
- 最近一次截图识别成功的关键状态
- 连续失败次数

这部分后面无论是接 UI，还是接远程控制，都会直接用到。

### 24.10 建议预留截图证据

长循环任务最有价值的调试能力之一，是在关键失败点自动留图。

建议至少考虑下面几种截图场景：

- 进入恢复流程前截图
- 超时前截图
- 连续失败达到阈值时截图
- 断线检测到时截图
- 检测到未知界面时截图

不一定一开始全做，但至少要把这个能力预留进设计里。

### 24.11 配置项建议单独分组

长循环任务建议配置不要只塞几个数字，最好按组管理：

- 基础配置
  - 窗口索引、任务入口、窗口模式
- 循环配置
  - 次数、间隔、时长、超时
- 恢复配置
  - 重试次数、恢复动作开关、重登开关
- 告警配置
  - 声音提醒、截图开关、人工暂停

这样后面 UI 配置弹窗也更容易做。

### 24.12 代码实现建议

如果后面真的开始做，我建议长循环任务优先按下面结构来：

- 任务类本身只表达状态流转
- 图片识别单独收口
- 点击动作尽量走统一封装
- 超时和重试不要散落在各个 `if` 里
- 恢复逻辑单独放成一组方法

可以先按下面这种职责拆分：

- `prepare()`
- `enter_task()`
- `run_once()`
- `recover()`
- `should_stop()`
- `after_round()`

### 24.14 对本项目最合理的落地方向

如果这个长循环任务是“一个明确业务任务反复执行”，我建议：

- 优先做成独立脚本
- 但尽量复用 `mhxy/__init__.py` 的基础能力
- 如果状态比较规则，再考虑接 `Common`

如果这个长循环任务是“多个任务轮流跑”，我建议：

### 24.15 当前阶段最值得先明确的事

在你正式规划这个任务时，我建议先把下面 5 件事定下来：

2. 失败后是强恢复继续跑，还是保守停机等待人工处理。 失败后继续跑其它的
3. 这个任务是否必须支持多窗口。 必须支持多窗口
4. 这个任务是否要接入现有 PyQt UI。 接入现有 PyQt UI
5. 这个任务的“完成”到底是按轮次、按时间还是按资源耗尽来定义。都需要支持

## 25. 基于当前结论的任务设计草案

基于 24.15 已经明确的 5 个结论，这个长循环任务不再适合做成“某个单脚本里再套一层 while True”，而应该按“多窗口任务调度器”来设计。

### 25.1 任务定位

这个任务的本质不是单个业务脚本，而是一个调度层能力：

- 面向多个窗口
- 调度多个子任务
- 某个子任务失败后不中断整体
- 支持接入现有 PyQt UI
- 支持多种完成条件

所以它更像：

- `Task Orchestrator`
- `Window Scheduler`
- `Mission Loop Runner`

而不是新的 `mhxy_xxx.py` 单任务脚本。

### 25.2 推荐落地形态

建议新增一个独立调度脚本，暂时不要直接堆进 `mhxy_richang.py`。

建议形态：

- 新建一个调度入口脚本
  - 例如：`mhxy_scheduler.py`
- UI 通过这个入口脚本启动调度
- 调度脚本再去调用已有任务类或已有任务脚本

这样做的好处是：

- 不污染现有单任务脚本
- 便于逐步替换老的串行入口
- 失败时更容易知道是“调度层问题”还是“子任务问题”

### 25.3 推荐职责拆分

这个长循环任务建议至少拆成 4 层职责。

#### A. 调度配置层

负责表达：

- 哪些窗口参与调度
- 每个窗口跑哪些任务
- 每个任务的启用状态
- 完成条件
- 失败后的策略
- 是否允许恢复

#### B. 窗口运行上下文层

负责表达每个窗口自己的运行态：

- 窗口索引
- 当前窗口状态
- 当前运行任务
- 已完成轮次
- 连续失败次数
- 最近成功时间
- 是否暂停

这一层最好独立成对象，不要继续共用全局状态概念。

#### C. 任务执行适配层

负责把“调度器”与“现有任务脚本”接起来。

建议统一一层适配接口，例如：

- `prepare(window_ctx)`
- `run(window_ctx)`
- `check_done(window_ctx)`
- `recover(window_ctx)`

这样后面无论子任务是：

- 复用已有类
- 调用已有脚本
- 新写状态机任务

都能挂到同一套调度逻辑里。

#### D. UI 展示层

负责：

- 开始调度
- 停止调度
- 查看每个窗口当前任务
- 查看最近失败原因
- 查看整体进度

UI 不应该直接承载调度逻辑，只做控制和展示。

### 25.4 建议的调度模型

结合你的要求，推荐采用“窗口独立调度 + 全局统一管理”的模式。

核心思想：

- 每个窗口有自己的任务队列和运行状态
- 每个窗口失败后不拖死其它窗口
- 调度器只负责决定“下一个该跑谁”

建议模型：

- 全局调度器
  - 管理所有窗口
- 窗口会话
  - 每个窗口一份状态
- 任务单元
  - 每个任务有自己的执行和完成判断

### 25.5 失败策略建议

你已经明确“失败后继续跑其它的”，这意味着失败策略不能只是 `sys.exit()` 或停止整个进程。

建议把失败分成 3 类：

- `soft_fail`
  - 当前任务失败，但窗口还能继续下一个任务
- `recoverable_fail`
  - 先做恢复动作，再决定是否继续
- `hard_fail`
  - 当前窗口暂停，但不影响其它窗口

默认策略建议：

- 单任务失败：记录失败并切到下一个任务
- 连续失败达到阈值：暂停当前窗口
- 所有窗口都不可继续：整体结束

### 25.6 完成条件建议

你已经明确“轮次 / 时间 / 资源耗尽都需要支持”，所以完成条件不要写死在各个任务里，应该配置化。

建议统一抽象为 3 组条件：

- `round_limit`
  - 达到指定轮次后任务完成
- `time_limit`
  - 到达指定运行时长或截止时间后任务完成
- `resource_limit`
  - 某项资源耗尽后任务完成

并支持组合策略：

- 任一满足即完成
- 全部满足才完成

更推荐默认使用：

- 任一满足即完成

因为这更符合自动化任务的安全停止原则。

### 25.7 多窗口支持的实现建议

多窗口是硬要求，但在当前项目里也是风险最高的一项。

建议不要一上来做真正并行线程调度。

第一阶段建议：

- 先做单进程、轮询式、多窗口串行调度
- 一次只激活一个窗口
- 每个窗口轮到时重新初始化上下文

理由：

- 当前底层大量依赖全局 `frame/frameSize/frameSizeCm`
- 真并发会非常容易互相污染
- 串行轮询更适合在现有代码基础上落地

也就是说：

- “支持多窗口”不等于“同时并行点击多个窗口”
- 更现实的方案是“调度器轮流服务多个窗口”

### 25.8 PyQt 接入建议

你已经确定要接现有 PyQt UI，所以建议在 UI 层新增一个独立区域，而不是把逻辑混进已有零散按钮。

建议最少增加这些 UI 能力：

- 调度任务配置区
- 目标窗口选择区
- 每个窗口的当前状态展示
- 总开关
- 单窗口暂停/恢复
- 调度日志视图

第一阶段不一定要做复杂界面，但至少要支持：

- 选择任务组合
- 选择窗口范围
- 启动
- 停止
- 看到当前跑到哪一步

### 25.9 调度任务配置建议

建议新增一个独立配置文件，例如：

- `resources/scheduler/scheduler.ini`

或者更适合结构化表达的：

- `resources/scheduler/scheduler.json`

如果继续沿用 ini，建议最少包含这些内容：

- 窗口列表
- 启用任务列表
- 每个任务的完成条件
- 每个任务的失败策略
- 恢复开关
- 全局运行时长
- 每窗口最大连续失败次数

如果要表达复杂任务队列，其实 JSON 会比 ini 更合适。

### 25.10 推荐的数据结构

后续实现时，建议至少抽出下面几个概念对象：

- `SchedulerConfig`
- `WindowSession`
- `TaskSpec`
- `TaskResult`
- `SchedulerState`

建议含义：

- `SchedulerConfig`
  - 总体配置
- `WindowSession`
  - 单窗口会话状态
- `TaskSpec`
  - 一个任务的定义
- `TaskResult`
  - 单次执行结果
- `SchedulerState`
  - 整体调度状态

### 25.11 推荐的执行顺序

建议调度循环用下面的顺序：

1. 检查全局停止条件
2. 遍历窗口列表
3. 跳过已暂停或已完成窗口
4. 选择该窗口当前应执行的任务
5. 执行任务
6. 记录结果
7. 根据结果决定：
   - 进入下一任务
   - 重试
   - 恢复
   - 暂停该窗口
8. 进入下一窗口

这套顺序最重要的点是：

- 调度器只关心“选任务、拿结果、更新状态”
- 不把具体任务细节写进调度主循环

### 25.12 第一版最小可用范围

为了降低风险，我建议第一版只做 MVP，不要一次把所有理想能力做满。

建议第一版只支持：

- 多窗口串行调度
- 2 到 3 个现有任务接入
  - 例如：宝图、秘境、师门
- 失败后跳过当前任务
- 简单恢复
  - 例如：重新点击任务追踪、关闭通用弹窗
- 按轮次/时间停止
- UI 可启动和停止

第一版先不要做：

- 真正的并发多线程点击
- 远程控制接入
- 复杂优先级调度
- 自动重登全闭环
- 特别复杂的资源耗尽判断

### 25.13 推荐的开发顺序

基于你的规划，后续真正开工时建议按这个顺序来：

1. 定义调度器配置结构
2. 定义 `WindowSession` / `TaskResult` 这些核心对象
3. 做命令行版调度器原型
4. 先接 1 个窗口 + 2 个任务跑通
5. 扩展到多窗口串行调度

7. 再接 PyQt UI
8. 最后补监控信息和截图证据

这样能避免一开始 UI、调度、业务脚本三头一起改，导致很难排查问题。

### 26.2 完成条件判断方式

你已经明确：

这对第一版非常有利，因为说明调度器可以优先围绕“图片状态驱动”来做统一抽象。

建议把完成判断统一成 3 类：

- `done_by_image`
  - 出现某张完成图片即视为完成
- `done_by_timeout`
  - 超过该任务最大运行时长即视为完成或退出
- `done_by_external_limit`
  - 到达轮次、到达时间、资源耗尽

第一版建议默认每个任务都至少具备：

- 一个主完成图片

这样即使图片漏检，也不至于永远卡死。

### 26.3 我推荐的恢复动作

你这次把“恢复动作”交给我推荐，我建议按风险和通用性分 3 级。

#### 一级恢复：通用轻恢复

这类动作可以默认开启，风险较低：

- 关闭通用弹窗
- 重新点击任务追踪
- 重新点击活动入口
- 再次点击“继续/确认/领取”
- 再次触发自动战斗

适用场景：

- 任务流程没推进
- 弹窗挡住界面
- 某一步误停在中间状态

#### 二级恢复：窗口级恢复

这类动作建议按任务开关控制：

- 重新初始化窗口上下文
- 重新激活窗口
- 重新切到小窗/原窗模式
- 回到主界面后重新进入任务

适用场景：

- 窗口焦点丢失
- 坐标状态漂移
- 某个任务中途退出到普通场景

#### 三级恢复：会话级恢复

这类动作不要默认频繁触发：

- 检测断线并重登
- 停止当前任务并切换到下一个任务
- 暂停当前窗口，等待后续人工处理

适用场景：

- 连续失败达到阈值
- 检测到未知界面
- 检测到断线或登录态失效

### 26.4 第一版推荐恢复策略

对于第一版，我建议采用保守但能持续运行的策略：

- 单次识别失败
  - 重试几次
- 多次识别失败
  - 执行一级恢复
- 一级恢复后仍失败
  - 执行二级恢复
- 二级恢复后仍失败

- 当前窗口连续多个任务都失败

这个策略和你的要求一致：

- 失败后不拖死整体

- 多窗口整体继续运行
